    来源：操作系统 / note/概述/概述.md

    操作系统的运行环境¶

    程序的运行不仅依靠自身，还需要操作系统、计算机硬件（如CPU、内存、硬盘、显卡等）以及用户的共同参与 。为了实现保护与控制，硬件必须将操作系统与用户程序隔离 。

    处理器运行模式¶

    处理器构成¶

    处理器由运算器、控制器、一系列的寄存器以及高速缓存构成

    处理器通过程序状态字（PSW）等控制和状态寄存器来划分和记录运行状态 。

    • 内核态 (Kernel Mode)：用于运行操作系统程序，可以执行修改状态字、允许/禁止中断等“特权指令” 。

    • 用户态 (User Mode)：用于运行用户程序，只能执行“非特权指令” 。

    • 状态切换：内核态切换到用户态可以通过设置PSW实现 ；而用户态进入内核态的唯一途径是中断机制 。

    • 中断机制

    中断机制是现代计算机系统的核心，保障软硬件配合协同 。

    • 硬中断：外部事件引起，用于支持CPU与设备通信（如I/O操作、显卡通知显示完毕） 。它具有被动、随机出现且程序无感知的特点 。

    • 异常：CPU执行指令时自身出现的问题，如缺页异常、算术溢出、内存越界等 。

    • 软中断：通过执行特定指令（如系统调用）人为主动触发，以确定的方式从用户态进入核心态 。具有主动、确定的特点 。

    • 处理流程：硬件接收中断信号并保护现场（保存寄存器到栈） -> 查中断向量表 -> 将处理程序入口地址推入寄存器 -> 执行中断处理程序 -> 恢复中断断点并继续执行原程序 。

    • 系统调用

    • 系统调用是操作系统提供给应用程序的接口，是程序唯一可以主动进入操作系统（特权模式）的操作方式 。

    • 与函数库调用的区别：函数库调用属于过程调用，开销小且处理器模式不变（例如libc中约有300个函数）；而系统调用需要利用软中断（如TRAP、INT指令），在用户和内核上下文间切换，开销较大，且不同操作系统的系统调用各不相同（例如UNIX约有90个） 。

    • PSP (Process Stack Pointer)： 进程堆栈指针，通常用于用户级的线程或任务。这种设计增强了系统的稳定性。

    • 程序计数器 (PC)： 在执行过程中，PC 的读出值通常是当前指令地址加上 4（因为指令都是32位，4个字节）。

    控制寄存器 (CONTROL) ​这个寄存器决定了 CPU 的“身份”和“装备”。它主要有两位： ​SPSEL (Bit 1): 堆栈选择 ​0: 使用主堆栈指针 (MSP)。 ​1: 使用进程堆栈指针 (PSP)。 ​注：在中断处理程序中，此位固定为 0（强制使用 MSP）。 ​nPRIV (Bit 0): 特权级控制 ​0: 特权级（可以访问所有资源）。 ​1: 用户级（受限制，不能直接修改特殊寄存器，不能触发某些系统操作）。

    1. 核心机制：执行优先级 vs 寄存器数值 ​在 Cortex-M3 中，处理器的状态由“模式”和“特权级”共同决定： ​Handler Mode（处理模式）： 只要在中断里，就一定处于这个模式。这个模式强制具备“特权级”，无论 CONTROL[0] 是什么。 ​Thread Mode（线程模式）： 正常运行主程序时处于此模式。此时是否具备特权，由 CONTROL[0] 决定。 ​关键结论： ​当一个在“用户级”（CONTROL[0]=1）运行的任务被中断触发时： ​硬件自动切换到 Handler Mode。 ​在 Handler Mode 下，CPU 无视 CONTROL[0] 的设置，强制以“特权身份”运行，这样它才能访问受保护的外设和特殊寄存器。 ​但是，CONTROL[0] 寄存器里的那个 1 可能依然留在那里，并没有被硬件擦除。

    第二部分：操作系统的关键概念

    1. 进程 (Process)

    2. 定义：运行中的程序，是所有操作系统的核心概念 。

    3. 本质：可以看作是一个“容器”，掌握着程序运行的上下文（如寄存器、堆栈、指令指针）并与内存地址、资源集合及用户号相关联 。

    4. 产生原因：为了支持多道程序设计，保存系统中多个并发执行程序的运行信息 。进程由其他进程创建，呈现树状结构 。

    5. 地址空间、文件与输入输出

    6. 地址空间：进程“看到”的一段虚拟内存区域，它可能大于或小于真实物理内存，使用时需要将虚拟地址转换为物理地址 。

    7. 文件：是一种长期或临时存储设备中的数据流抽象模型，独立于具体物理设备，通常以树状层次结构的目录进行组织和访问 。

    8. 输入/输出 (I/O)：由操作系统通过驱动程序管理，驱动程序通常由硬件厂商开发并运行在内核态 。

    9. 内核架构 (Kernel)

    内核是操作系统的核心部分，负责管理进程、内存、文件和系统资源等 。常见的内核结构包括：

    • 宏内核 (Monolithic Kernel)：整个内核程序是一个单一二进制可执行文件，大部分UNIX和Linux采用此结构 。

    • 微内核 (Microkernel)：仅将最核心的功能（如基本线程管理、内存管理、IPC）加入内核，如Mach、Minix系统 。

    • 混合内核 (Hybrid Kernel)：结合了宏内核与微内核的特点，如Windows NT 。

    第三部分：操作系统的四大特征

    1. 并发：宏观上多个程序同时执行；微观上任何时刻只有一个程序在CPU上执行（轮流执行，与并行有区别） 。

    2. 共享：操作系统与多个用户的程序共同使用计算机系统中的有限资源，需解决资源分配与保护问题 。

    3. 虚拟：将一个物理实体映射为若干个逻辑实体，使得逻辑上每个进程仿佛都有独占的CPU和内存空间 。

    4. 随机：多个进程“走走停停”，推进速度不可预知，且必须随时响应不可预测的外部事件或中断 。

    进程¶

    CPU的利用率

    1. 核心定义：什么是进程？¶

    简单来说，进程是正在运行的程序的实例。

    • 程序（Program） 是静态的。它只是存储在硬盘上的一堆编译好的代码和数据（比如一个 .exe 文件或你在 macOS 终端里跑的脚本）。

    • 进程（Process） 是动态的。当操作系统把程序加载到内存中，并分配了 CPU 资源让它开始执行时，它就变成了一个进程。

    如果把“程序”比作一张菜谱，“进程”就是厨师阅读菜谱、准备食材、并在厨房里真正炒菜的全过程。

    2. 进程的内部结构（内存视图）¶

    一个标准的进程内存空间通常从低地址到高地址分为以下几个部分：

    • 代码段（Text Section）： 存放编译后的机器指令（你的代码）。这部分通常是只读的，防止程序意外修改自己的指令。

    • 数据段（Data Section）： 存放全局变量和静态变量。

    • 堆（Heap）： 用于动态内存分配。当你在代码里使用类似 malloc（C语言）或 new（C++）时，内存就是从这里申请的。堆向高地址增长。

    • 栈（Stack）： 用于存放局部变量、函数参数和返回地址。每次调用函数时，相关数据会被压入栈中；函数返回时弹出。栈向低地址增长。

    3. 进程控制块 (PCB)¶

    操作系统可能同时运行成百上千个进程，为了管理它们，内核为每个进程维护了一个数据结构，称为 进程控制块（Process Control Block, PCB）

    在底层实现中，创建一个进程（或任务）本质上就是分配内存空间并初始化这个控制块。PCB 包含了操作系统管理该进程所需的所有关键信息：

    • 进程状态（State）： 进程当前是在运行、就绪还是阻塞？

    • 程序计数器（PC）： 记录进程下一次要执行的指令地址。

    • CPU 寄存器： 当进程被暂停时，需要保存当前 CPU 寄存器的状态，以便将来恢复执行。

    • 内存管理信息： 页表或段表，记录该进程使用了哪些内存。

    • 记账信息与 I/O 状态： 进程打开了哪些文件，占用了多少 CPU 时间等。

    4. 进程的生命周期与状态转换¶

    进程从创建到消亡，会经历不同的状态。最经典的五状态模型如下：

    • 创建（New）： 进程正在被创建，操作系统正在为其分配 PCB 和资源。

    • 就绪（Ready）： 进程已经准备好运行，万事俱备，只欠 CPU 调度。它在就绪队列中排队。

    • 运行（Running）： 进程获得了 CPU，正在执行它的指令。

    • 阻塞/等待（Blocked/Waiting）： 进程因为等待某个事件发生（比如等待用户输入、等待网络数据、等待磁盘读取）而暂停。此时即使给它 CPU，它也无法运行。

    • 终止（Terminated）： 进程执行完毕或被系统强行杀死，操作系统回收其占用的所有资源。

    5. 核心动作：上下文切换(Context Switch)¶

    单个 CPU 核心在同一时刻只能运行一个进程。操作系统通过上下文切换制造出多个进程“同时”运行的假象（并发）。

    1. 保存现场： 当操作系统决定暂停当前运行的进程 A 时，会将 CPU 当前的所有状态（程序计数器、各种通用寄存器的值、栈指针等）保存到进程 A 的 PCB 中。

    2. 调度： 调度器从就绪队列中挑选下一个要运行的进程 B。

    3. 恢复现场： 操作系统将进程 B 的 PCB 中保存的寄存器值和程序计数器加载回 CPU 中。

    4. 继续执行： CPU 接着执行进程 B 的代码。

    这种切换往往需要借助底层的系统调用或硬件中断来触发。例如，通过特定的汇编指令触发软中断，进入内核态，完成环境的保存与切换，然后再跳转到新任务的第一条指令开始运行。

    1. 线程的“交织”执行 (Interleaving)¶

    观察输出顺序，你会发现它并不是“主线程创建完所有 10 个线程，然后子线程才开始打印”。

    • 现象： 在 Main here.Creating thread 1 之后，紧接着就出现了 Hello World! Greeting from thread #0!。

    调度¶

    当计算机系统是多道程序设计系统时，通常就会有多个进程或线程同时竞争CPU。只要有两个或更多的进程处于就绪状态，这种情形就会发⽣。如果只有⼀个CPU可⽤，那么就必须选择下⼀个要运⾏的进程。在操作系统中，完成选择⼯作的这⼀部分称为调度程序（scheduler），该程序使⽤的算法称为调度算法（schedulingalgorithm）。

    进程可以分成两个类型

    • 计算密集型

    • I/O密集型

    非抢占式：进程可以一直执行

    抢占式：进程可以被操作系统打断

    抢占式调度算法挑选一个进程，并且让该进程运行某个固定时段的最大值。如果在该时段结束时，该进程仍在运行，它就被挂起，而调度程序挑选另一个进程运行（如果存在一个就绪进程）。进行抢占式调度处理，需要在时间间隔的末端发生时钟中断，以便把CPU控制返回给调度程序。如果没有可用的时钟，那么非抢占式调度就是唯一的选择了。

    调度算法的目标¶

    1. 所有系统¶

    无论什么类型的系统，调度程序都应满足：

    • 公平：给每个进程合理的 CPU 份额，避免某个进程被无限期“饿死”。

    • 策略强制执行：确保系统设定的规则（如：安全控制进程优先于发薪水进程）得到彻底执行。

    • 平衡：尽可能让系统的所有部分（CPU、I/O 设备）都忙碌起来，不出现闲置。

    2. 不同环境下的特定目标¶

    调度算法需要根据应用场景进行权衡（Trade-off）：

    系统类型 核心目标 说明
    批处理系统 吞吐量、周转时间 追求单位时间内完成作业数最大化，以及减少作业从提交到完成的总时间。
    交互式系统 响应时间、均衡性 重点在于快速响应用户的操作，并符合用户的心理预期（均衡感）。
    实时系统 截止时间、预测性 必须在限定时间内完成任务（避免丢数据），且系统的行为必须是可预测的。

    3. 批处理系统的三大衡量指标¶

    文中详细解释了管理大型计算中心时常用的三个概念：

    1. 吞吐量 (Throughput)：系统每小时完成的作业数量。越大越好。

    2. 周转时间 (Turnaround Time)：从作业提交到完成的平均等待时间。越小越好。

    3. 注意：吞吐量和周转时间有时是矛盾的。只运行短任务可以提高吞吐量，但会导致长任务的周转时间变得无限长。

    4. CPU 利用率 (CPU Utilization)：CPU 忙碌的时间比例。

    5. 观点：作者认为这不是一个好的度量参数，就像通过引擎转速来评价汽车好坏一样不可靠；真正有价值的是系统实际完成了多少工作。

    4. 调度策略的优化技巧¶

    • 混合搭配：在内存中同时保留 CPU 密集型和 I/O 密集型进程。

    • 原因：如果全是 CPU 密集型，磁盘会闲置；如果全是 I/O 密集型，CPU 会闲置。通过组合，可以让 CPU 和磁盘同时运转，提高整体效率。

    一句话总结：

    没有完美的调度算法，只有最适合环境的算法——批处理看效率（吞吐量），交互式看速度（响应时间），实时系统看准时（截止时间）。

    关键时间定义

    • 到达时间

    • 进程进入就绪队列的时间

    • 完成时间

    • 进程完成的时间

    • 周转时间

    • 周转时间=完成时间-到达时间

    • 处理时间

    • 进程在CPU上执行的时间

    • 等待时间

    • 等待时间=处理时间-周转时间

    • 响应时间

    • 到达系统到首次获得CPU的时间

    批处理系统中的调度¶

    先来先服务（FCFS）¶

    按照进程请求CPU的顺序运行，非抢占式

    缺点在于长进程后面的短进程要一直等待

    最短作业优先(SJF)¶

    假设可以精确预测每个作业运行时间

    假设作业之间相互独立、同时运行

    最短剩余时间优先¶

    当一个新就绪的进程比当前运行进程具有更短的运行时间，就抢占

    优点在于拥有最短的平均周转时间

    缺点在于长时间的任务迟迟得不到解决，产生饥饿现象

    最高响应比优先¶

    计算响应比

    $R=\frac{周转时间}{处理时间}$
